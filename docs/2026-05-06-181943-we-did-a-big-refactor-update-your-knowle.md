## Strengths
<!-- What already exists that supports this goal? What's solid? -->

- The refactor produced a clearer multi-project shape: `HappyGymStats.Api`, `HappyGymStats.Core`, `HappyGymStats.Data`, `HappyGymStats.Contracts`, `HappyGymStats.Identity`, `HappyGymStats.Encryption`, `HappyGymStats.AdminPanel`, and the new Blazor host/client split. This is visible in `HappyGymStats.sln` and the `src/` project layout.
- The production routing design is simple and appropriate for the frontend/backend problem: `infra/nginx-torn.conf:17` proxies `/api/` to `127.0.0.1:5047`, while `infra/nginx-torn.conf:27` proxies the Blazor app to `127.0.0.1:5182`. The API and Blazor service files align with those ports at `infra/happygymstats-api.service:15` and `infra/happygymstats-blazor.service:15`.
- The Blazor frontend has a single typed backend boundary: `src/HappyGymStats.Blazor/HappyGymStats.Blazor/Program.cs:13` reads `ApiBaseUrl`, `Program.cs:14` registers `HttpClient<SurfacesService>`, and `src/HappyGymStats.Blazor/HappyGymStats.Blazor/Services/SurfacesService.cs:10` / `:18` centralize surfaces and import calls. That makes the 502 path narrow to diagnose.
- The surfaces API itself is small and deterministic: `src/HappyGymStats.Api/Controllers/SurfacesController.cs:6` owns `/api/v1/torn/surfaces`, and the `latest` endpoint reads the generated `latest.json` file rather than rebuilding data on each request. Missing cache data returns a structured 404, not a thrown exception.
- The API boot path creates the surfaces cache directory on startup (`src/HappyGymStats.Api/Program.cs:35`, `:37`) and registers `SurfacesCacheWriter` plus `ImportOrchestrator` (`src/HappyGymStats.Api/Program.cs:66`-`:74`), so import completion should refresh the frontend dataset without a separate manual export step.
- The deployment scripts now use a consistent timestamped release/symlink pattern across backend, frontend, and admin panel. `scripts/deploy-adminpanel.sh:19`-`:22` mirrors the release/current/staging pattern used by `scripts/deploy-backend.sh` and `scripts/deploy-frontend.sh`.
- The local sudoers file already contains most of the commands the deploy scripts need for release activation and service restarts, including `rsync`, `mkdir`, `rm`, `ln`, `chown`, `chmod`, and restarts/status for API, Blazor, and AdminPanel (`infra/sudoers-happygymstats:1`-`:24`).
- The AdminPanel app is intentionally separated from the main API and has an explicit read-only posture: `src/HappyGymStats.AdminPanel/Program.cs` registers its own DB context with `UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking)` and does not register `IdentityMap` access for resolving player identity.
- The AdminPanel service file exists locally and uses a dedicated port and deployment root: `infra/happygymstats-adminpanel.service:9`-`:15` points to `/var/www/happygymstats-adminpanel/current` on `127.0.0.1:5048`.
- There is useful verification infrastructure around surfaces generation: `scripts/verify/s05-local-surfaces.sh` starts the API with `--no-launch-profile`, pins `HAPPYGYMSTATS_SURFACES_CACHE_DIR`, runs an import, waits for `meta.json`/`latest.json`, and validates required JSON keys. This is directly relevant to separating “API unreachable” from “cache not generated.”

## Gaps
<!-- What's missing, incomplete, or problematic relative to this goal? Be specific: file paths, patterns, missing abstractions. -->

- The likely frontend 502 cause is deployment/configuration, not the Blazor component itself. `src/HappyGymStats.Blazor/HappyGymStats.Blazor/appsettings.json:9` points `ApiBaseUrl` to `https://torn.geromet.com`, and `SurfacesService` calls absolute paths under that base (`Services/SurfacesService.cs:10`, `:18`). A 502 from those calls means nginx cannot get a good response from the upstream API at `127.0.0.1:5047`, or the Blazor server is configured with a different/bad `ApiBaseUrl` at runtime. The app currently surfaces only `Response status code does not indicate success: 502 (Bad Gateway)` from `EnsureSuccessStatusCode` (`Services/SurfacesService.cs:12`, `:19`), which hides upstream identity: API down, nginx route missing, bad port, TLS/proxy issue, or remote base URL mismatch.
- The frontend calls the public nginx URL from server-side Blazor instead of using the loopback API service URL. Because `@rendermode InteractiveServer` is used in `Home.razor`, the `SurfacesService` `HttpClient` runs server-side. Calling `https://torn.geromet.com/api/...` from the same VPS goes through Cloudflare/nginx/public TLS rather than directly to `http://127.0.0.1:5047`. That is workable, but more fragile than using an internal production `ApiBaseUrl` such as `http://127.0.0.1:5047` for server-side calls while still keeping browser-origin API routes available through nginx.
- There is no deployment-time health gate after restart. `scripts/deploy-backend.sh:48`, `scripts/deploy-frontend.sh:48`, and `scripts/deploy-adminpanel.sh:48` restart services but do not check `systemctl status`, call `/api/v1/torn/health`, call `/api/v1/torn/surfaces/latest`, or fail fast when nginx returns 502. This allows a broken API service, bad connection string, missing migration, or port conflict to ship silently and only appear in the UI.
- The API startup path can fail hard before any endpoint is available. `src/HappyGymStats.Api/Program.cs:81`-`:82` runs EF migrations at startup, and `AppConfiguration.ResolveConnectionString` throws if the production connection string is absent. If the server lacks a valid Postgres connection string or containers are not running, systemd will restart/fail the API and nginx will return 502 to Blazor. The current deploy scripts do not validate the database/container precondition before restarting API.
- The surfaces cache paths are split between nginx static cache and API dynamic cache without an explicit production contract. `infra/nginx-torn.conf:10` aliases `/data/surfaces/` to `/var/www/happygymstats/data/surfaces-cache/`, while `AppConfiguration.ResolveSurfacesCacheDirectory` falls back to `<contentRoot>/data/surfaces` in production (`src/HappyGymStats.Api/Infrastructure/AppConfiguration.cs:25`). Unless `HAPPYGYMSTATS_SURFACES_CACHE_DIR` is set in the API systemd environment, static `/data/surfaces/` and API `/api/v1/torn/surfaces/latest` may point at different directories. That mismatch can cause stale/missing static artifacts even when the API endpoint works.
- The service files do not set critical runtime environment variables. `infra/happygymstats-api.service` only sets `ASPNETCORE_ENVIRONMENT` and `ASPNETCORE_URLS`; it does not set `HAPPYGYMSTATS_CONNECTION_STRING`, `ConnectionStrings__HappyGymStats`, `HAPPYGYMSTATS_SURFACES_CACHE_DIR`, or `ProvisionalToken__SigningKey`. `src/HappyGymStats.Api/appsettings.json:10` contains a local `changeme` Postgres connection string and `:13` contains a placeholder signing key. Relying on checked-in placeholders in production is risky and likely to produce failed startup or insecure tokens.
- The AdminPanel deployment script publishes and flips releases but does not install its service. `scripts/deploy-adminpanel.sh` restarts `${DEPLOY_ADMINPANEL_SERVICE}` at `:48`, but nothing in the script copies `infra/happygymstats-adminpanel.service` into `/etc/systemd/system/`, runs `systemctl daemon-reload`, enables the service, or starts it if it does not already exist. This matches the reported “service file needs to be placed and enabled” gap.
- The AdminPanel deployment path depends on a local sudoers file that the script never installs. `infra/sudoers-happygymstats:1`-`:24` has the updated NOPASSWD permissions locally, but no script copies it to `/etc/sudoers.d/`, validates it with `visudo -cf`, or atomically installs it. If the server still only allows `/usr/bin/rsync`, `scripts/deploy-adminpanel.sh:36`-`:43` will fail on `mkdir`, `ln`, `chown`, `find`, or `chmod` before it reaches restart.
- The local sudoers file is still not sufficient for the intended one-time setup script. It allows `systemctl restart/status` but not `systemctl enable`, `systemctl start`, `systemctl daemon-reload`, `nginx -t`, `systemctl reload nginx`, `install`, `tee`, or `visudo`. If the one-time setup needs to place service/sudoers/nginx files safely, those exact command allowances must be designed narrowly rather than added ad hoc.
- There is no nginx route for AdminPanel yet. `infra/happygymstats-adminpanel.service:15` exposes `127.0.0.1:5048`, but neither `infra/nginx-torn.conf` nor `infra/nginx-auth.conf` proxies an admin host/path to 5048. `docs/v2-plan.md` covers auth/keycloak nginx setup (`:57`-`:58`, `:131`) but not an admin panel host or route. Even if the service is installed and running, it may not be externally reachable.
- AdminPanel authorization likely needs a deployment smoke test before exposure. `src/HappyGymStats.AdminPanel/Program.cs` uses fallback authorization requiring the `admin` role, while `AdminHealthController` is anonymous at `/admin/health`. That is a good split, but there is no deploy script or documented check for `/admin/health`, protected `/admin/api/v1/import-runs`, Keycloak issuer reachability, or role mapping.
- Documentation is stale after the refactor. `README.md` still describes SQLite and a static dashboard; `docs/OVERVIEW.md` says the core flow persists results in SQLite and is consumed by `web/`; `docs/SETUP.md` documents `HAPPYGYMSTATS_DATABASE` and SQLite paths; `docs/DEPLOYMENT.md` says frontend uploads `web/` to `/var/www/torn-frontend/current`, while the actual deployment publishes Blazor to `/var/www/happygymstats-blazor`. These stale docs will mislead future fixes for the 502 issue.
- API sample requests are stale. `src/HappyGymStats.Api/HappyGymStats.Api.http` uses `/v1/health`, `/v1/gym-trains`, `/v1/import`, etc., but controllers now route under `/api/v1/torn/...`. This weakens manual smoke testing and can send operators to false failures.
- Tests still depend heavily on SQLite despite the production move to PostgreSQL. `tests/HappyGymStats.Tests/HappyGymStats.Tests.csproj:12` references `Microsoft.EntityFrameworkCore.Sqlite`, and `tests/HappyGymStats.Tests/ApiEndpointTests.cs:45` asserts `Sqlite` as the health provider. SQLite in-memory tests are useful for fast unit coverage, but they will not catch production Npgsql/migration/startup failures that produce nginx 502s.
- There is no end-to-end deployment smoke script that exercises the real server shape: nginx → Blazor → API → Postgres → surfaces cache. `scripts/verify/s05-local-surfaces.sh` is strong locally, but it cannot detect bad systemd unit env, missing `/etc/sudoers.d` install, missing AdminPanel service registration, Cloudflare/nginx upstream failures, or stale production cache paths.
- Several scripts are not aligned with the new “agent can execute safely” standard. `scripts/server-create-containers-user.sh` prompts interactively for secrets and tells a human to run it; `scripts/deploy-containers.sh` uses hardcoded SSH settings instead of shared `deploy-config.sh`; `scripts/recon-server.sh` ends with “Paste full output back to Claude.” Those are operationally useful but not yet polished for repeatable unattended deployment.
- Package references use loose or very new versions in places. The Blazor projects use `MudBlazor` `8.*`, and the solution targets `net10.0`/EF Core 10.0 packages. That may be intentional, but it increases deployment reproducibility risk unless the server SDK/runtime and package lock strategy are explicit.

## Next Steps

1. **Add production health gates to deployment.** After API restart, verify `systemctl status happygymstats-api`, `http://127.0.0.1:5047/api/v1/torn/health`, and `https://torn.geromet.com/api/v1/torn/health`; after Blazor restart, verify `systemctl status happygymstats-blazor` and page reachability; after AdminPanel setup, verify `http://127.0.0.1:5048/admin/health` and the external nginx route if added. Make deploy fail on bad status instead of leaving a 502 for users.
2. **Fix production API configuration explicitly.** Add a deployment contract for `HAPPYGYMSTATS_CONNECTION_STRING`/`ConnectionStrings__HappyGymStats`, `ProvisionalToken__SigningKey`, and `HAPPYGYMSTATS_SURFACES_CACHE_DIR`; wire these through systemd environment files or secure server-local env files, not checked-in placeholder appsettings. Align API cache directory with nginx `/data/surfaces/` if that static route remains supported.
3. **Decide and standardize Blazor’s production API base URL.** For server-side Blazor, prefer an internal base URL (`http://127.0.0.1:5047`) or a relative/same-origin backend handler that avoids public loopback through Cloudflare/nginx. If public same-origin remains the intended path, add explicit diagnostics that distinguish nginx 502 from API 404/no-cache and document the tradeoff.
4. **Create a one-time AdminPanel server setup script.** It should install `infra/happygymstats-adminpanel.service` to `/etc/systemd/system/`, run `systemctl daemon-reload`, `systemctl enable --now happygymstats-adminpanel`, install/update the required sudoers file via a safe temp file plus `visudo -cf`, and verify `/admin/health`. Keep it idempotent and narrow.
5. **Update sudoers as a deployment artifact, not a local note.** Design exact NOPASSWD entries for setup and steady-state deploy separately. Steady-state should allow only release activation and service restart/status; one-time setup may temporarily require `daemon-reload`, `enable`, `start`, `install`, `visudo`, and possibly `nginx -t`/reload if AdminPanel nginx is also installed.
6. **Add AdminPanel nginx routing.** Decide whether AdminPanel lives under a separate host (for example `admin.geromet.com`) or a path. Add a 5048 upstream config with auth/TLS expectations, then include nginx validation and reload in the setup flow.
7. **Add a production smoke script.** It should check local ports, systemd units, nginx config, external `/api/v1/torn/health`, external `/api/v1/torn/surfaces/latest`, Blazor home load, AdminPanel `/admin/health`, and container health for Postgres/Keycloak. This should be safe to run after every deploy.
8. **Refresh docs after the refactor.** Update `README.md`, `docs/OVERVIEW.md`, `docs/SETUP.md`, `docs/DEPLOYMENT.md`, and `src/HappyGymStats.Api/HappyGymStats.Api.http` to reflect Blazor, PostgreSQL, Keycloak, current routes, current deploy roots, and the canonical import→reconstruct→surfaces flow.
9. **Add production-provider test coverage.** Keep fast SQLite tests if useful, but add a Postgres-backed integration path (containerized or Testcontainers) that applies migrations and hits `/api/v1/torn/health` plus surfaces endpoints. This catches the class of startup failures that become nginx 502s.
10. **Improve frontend error diagnostics.** Replace raw `EnsureSuccessStatusCode` UI messages with a small response classifier that reports status, endpoint, and likely next action without exposing secrets. Preserve structured logs server-side so operators can connect the visible failure to the upstream health check.
11. **Normalize deployment scripts.** Move container deploy scripts onto `deploy-config.sh`, remove hardcoded SSH duplication, avoid human-paste instructions in operational scripts, and make preconditions machine-checkable.
12. **Pin deployment/runtime versions.** Confirm net10 runtime availability on the server or switch to a supported pinned runtime; avoid loose package versions for deployed apps unless lockfiles or restore reproducibility are in place.
